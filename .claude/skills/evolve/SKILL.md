# 하네스 진화 스킬

> 실패를 기록하고, 패턴을 보고, 하네스를 고친다.

## Step 1: 실패 데이터 수집

### 1-1. 피드백 로그 읽기
- `docs/harness/feedback-log.md` 파일을 읽는다
- **상태: open** 인 항목만 추출한다

### 1-2. 데이터 없으면 종료
- open 상태인 항목이 없으면:
- "현재 분석할 실패 데이터가 없습니다." 출력 후 종료

## Step 2: 패턴 분석

수집된 실패 데이터를 분류한다:

### 분류 기준
| 패턴 | 의미 | 조치 방향 |
|------|------|-----------|
| 같은 도구에서 같은 위반이 N회 반복 | 규칙이 있는데 안 지켜짐 | CLAUDE.md 표현 보강 (예시 코드 추가 등) |
| 새로운 종류의 위반 | 규칙이 없음 | CLAUDE.md에 새 규칙 추가 |
| 가드레일에서 잡히지 않는 위반 | 가드레일 미비 | ArchUnit/Checkstyle/ESLint 규칙 추가 |
| No-touch 파일 수정 시도가 반복됨 | 컨텍스트 부족 | CLAUDE.md에 "왜 건드리면 안 되는지" 설명 보강 |
| 의존성 추가 시도가 반복됨 | 기존 라이브러리 안내 부족 | CLAUDE.md에 사용 가능한 라이브러리 목록 추가 |

### 출력 형식
```
## 피드백 분석 결과

### 발견된 패턴
1. [패턴 설명] — N회 발생
   - 원인: [규칙 부재 / 표현 부족 / 가드레일 미비]
   - 관련 로그: [날짜, 도구, 파일]

### 변경 제안
1. [어디를] [어떻게] 변경
   - 변경 이유: [왜 이 변경이 같은 실수를 방지하는지]
```

## Step 3: 변경 제안

분석 결과를 바탕으로 구체적인 변경을 제안한다:

- **CLAUDE.md 변경**: 규칙 추가/보강, 예시 코드 추가
- **backend/CLAUDE.md 변경**: 백엔드 특화 규칙 추가
- **frontend/CLAUDE.md 변경**: 프론트엔드 특화 규칙 추가
- **가드레일 변경**: ArchUnit 테스트 추가, Checkstyle 규칙 추가, ESLint 규칙 추가
- **Hook 변경**: 보호 파일 목록 추가/수정

**반드시 사용자에게 변경 내용을 보여주고 승인을 받은 후에 적용한다.**

## Step 4: 적용 + 정리 (승인 후)

### 4-1. 하네스 변경 적용
- 승인된 변경 사항을 실제 파일에 반영한다

### 4-2. 로그 정리
- 반영된 항목의 상태를 `open` → `resolved` 로 변경한다

### 4-3. 결과 요약
```
## 진화 완료

### 적용된 변경
- [변경 1]
- [변경 2]

### 정리된 로그
- N개 항목 resolved
```
